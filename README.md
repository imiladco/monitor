# Site Monitor

مانیتور اختصاصی سایت‌های وردپرسی/ووکامرسی که روی هاست‌های جدا هستن — با داشبورد وب و "Time Machine": تایم‌لاینی که دقیق نشون می‌ده کِی پلاگین/پوسته آپدیت شده، یوزر ادمین جدید ساخته شده، یا دیتابیس رشد غیرعادی داشته.

## اجزا

- **`server/`** — API مرکزی (Node.js/Express + SQLite): چک‌های بیرونی (آپ‌تایم، سرعت پاسخ، چک‌اوت، SSL) رو زمان‌بندی می‌کنه، snapshotهای پلاگین همراه رو دریافت و دیف می‌کنه، هشدار تلگرام می‌فرسته
- **`dashboard/`** — پنل وب (React) برای دیدن وضعیت زنده‌ی همه‌ی سایت‌ها و تایم‌لاین Time Machine هر کدوم
- **`wp-agent/`** — پلاگین سبک وردپرسی (mu-plugin) که تغییرات پلاگین/پوسته/یوزر/دیتابیس رو push می‌کنه. **اختیاریه** — بدون اون هم آپ‌تایم/سرعت/SSL کار می‌کنه، فقط Time Machine و رویدادهای امنیتی نیاز به agent دارن.

## راه‌اندازی

```bash
npm install
npx playwright install chromium   # برای اسکرین‌شات و Core Web Vitals لازمه
npm run build:dashboard           # داشبورد رو می‌سازه (dashboard/dist)
cp .env.example .env
cp config/sites.example.json config/sites.json
```

اگه به هر دلیلی نمی‌خوای/نمی‌تونی کرومیوم نصب کنی، `CHROMIUM_PATH` رو تو `.env` به مسیر یه کرومیوم/کروم نصب‌شده‌ی دیگه اشاره بده. بدون کرومیوم، فقط بخش اسکرین‌شات/Core Web Vitals غیرفعال می‌مونه — بقیه (آپ‌تایم، SSL، دامنه، agent) عادی کار می‌کنه.

### رمز عبور داشبورد (اجباری)

`ADMIN_PASSWORD` رو تو `.env` یه مقدار قوی بذار — بدونش کل پنل و API غیرفعال می‌مونن (بازم fail-closed می‌مونه، نه باز). این تنها رمزیه که لازمه؛ اضافه/ویرایش/حذف سایت و تنظیمات تلگرام همه از همین پنل با همین رمز انجام می‌شن، نه فایل.

### گروه تلگرام با تاپیک‌بندی (اختیاری)

به‌جای فرستادن همه‌ی هشدارها به یه چت، می‌تونی هر دسته (امنیت، وضعیت، افزونه، پوسته، دامنه، SSL، پرفورمنس) رو به یه تاپیک جدا توی یه گروه بفرستی:

۱. یه گروه بساز، ربات رو بهش اضافه کن و **ادمین** کن (با اجازه‌ی «Manage Topics»)
۲. توی تنظیمات گروه، **Topics** رو فعال کن
۳. یه پیام توی گروه بفرست (تا ربات چتش رو ببینه)
۴. از داشبورد → تنظیمات → «شناسایی خودکار گروه» رو بزن (یا Chat ID گروه رو دستی وارد کن)
۵. «ساخت خودکار همه‌ی تاپیک‌ها» رو بزن — ربات با Bot API خودش هفت‌تا تاپیک می‌سازه

اگه از قبل تاپیک ساخته بودی، تلگرام راهی برای لیست‌کردن تاپیک‌های قدیمی نمی‌ده — thread id همون تاپیک رو (از لینکش) دستی جلوی همون دسته وارد کن. کنار هر دسته دکمه‌ی «تست» هم هست.

بدون تنظیم گروه، همه‌چیز مثل قبل به همون یه چت/چت‌آیدی می‌ره.

### سایت‌ها و تلگرام — از پنل، نه فایل

`config/sites.json` فقط برای **اولین اجرا** استفاده می‌شه (اگه دیتابیس خالی باشه). بعد از اولین بار، سایت‌ها رو از داشبورد (دکمه‌ی «+ سایت جدید»، یا ویرایش/حذف تو صفحه‌ی هر سایت) مدیریت کن. توکن و chat id تلگرام هم از صفحه‌ی «تنظیمات» تو داشبورد ست می‌شن — نیازی به دست‌زدن به `.env` بعد از راه‌اندازی اولیه نیست.

### ۱. ربات تلگرام

با [@BotFather](https://t.me/BotFather) یه ربات بساز، توکن رو بگیر، بعد `chat.id` رو از `https://api.telegram.org/bot<TOKEN>/getUpdates` پیدا کن. هر دو رو تو `.env` بذار.

### ۲. تعریف سایت‌ها

`config/sites.json` رو با سایت‌های واقعی پر کن:

```json
[
  { "name": "Almasara Fast Cart", "url": "https://yoursite.com", "checkoutUrl": "https://yoursite.com/checkout/" }
]
```

### ۳. اجرا

```bash
npm start
```

داشبورد و API رو روی همون پورت بالا میاره (پیش‌فرض `http://localhost:4000`).

اولین بار که سرور بالا میاد، برای هر سایت یه **API key** توی لاگ چاپ می‌شه (برای اتصال agent) — یا از داخل داشبورد، توی صفحه‌ی هر سایت که هنوز agent وصل نشده، همون کلید نمایش داده می‌شه.

### اجرای دائمی روی VPS با PM2

```bash
npm i -g pm2
pm2 start ecosystem.config.cjs
pm2 save
pm2 startup
```

## نصب Site Monitor Agent روی هر سایت وردپرسی (اختیاری، برای Time Machine)

یه پلاگین معمولی وردپرسیه (نه mu-plugin). یه زیپ ازش بساز:

```bash
cd wp-agent && zip -r site-monitor-agent.zip site-monitor-agent
```

بعد مثل هر پلاگین دیگه نصبش کن:

۱. توی سایت وردپرسی: **افزونه‌ها ← افزودن افزونه‌ی جدید ← بارگذاری افزونه** → فایل زیپ رو انتخاب کن → نصب و فعال‌سازی
۲. برو **تنظیمات ← Site Monitor Agent**
۳. `API URL` رو بذار `https://your-vps-domain.com/api/ingest` (یا `http://IP:4000/api/ingest`)
۴. `API Key` رو از داشبورد (صفحه‌ی جزئیات همون سایت) کپی کن
۵. ذخیره کن، بعد دکمه‌ی «ارسال همگام‌سازی همین الان» رو بزن تا مطمئن بشی وصل شده

از این به بعد، هر آپدیت پلاگین/پوسته/هسته، هر یوزر ادمین جدید، رشد غیرعادی دیتابیس، و تغییر فایل‌های core خودکار توی تایم‌لاین اون سایت توی داشبورد ثبت می‌شه — و اگه حساس باشه (مثل ساخته‌شدن ادمین جدید یا دستکاری فایل core) به تلگرام هم هشدار می‌ره.

(برای نصب دستی/DevOps: می‌تونی به‌جای پر کردن فرم، `SITE_MONITOR_API_URL` و `SITE_MONITOR_API_KEY` رو به‌عنوان constant تو `wp-config.php` تعریف کنی — پلاگین این‌ها رو به فرم تنظیمات ترجیح می‌ده.)

## تنظیمات (`.env`)

| متغیر | توضیح |
|---|---|
| `TELEGRAM_BOT_TOKEN` / `TELEGRAM_CHAT_ID` | مقصد هشدارها |
| `PORT` | پورت داشبورد/API (پیش‌فرض ۴۰۰۰) |
| `CHECK_INTERVAL_MINUTES` | فاصله‌ی چک‌های بیرونی |
| `DAILY_SUMMARY_HOUR` | ساعت گزارش روزانه |
| `SSL_WARN_DAYS` | چند روز قبل انقضای SSL هشدار بده |
| `SLOW_RESPONSE_MS` | آستانه‌ی کند بودن پاسخ |
| `DB_GROWTH_WARN_PERCENT` | چند درصد رشد ناگهانی دیتابیس هشدار بسازه |

## چک‌های عمیق روزانه (visual / Core Web Vitals / دامنه)

هر روز ساعت `DEEP_CHECK_HOUR` (پیش‌فرض ۳ بامداد) برای هر سایت:

- یه اسکرین‌شات از هوم‌پیج می‌گیره و با اسکرین‌شات روز قبل مقایسه می‌کنه — اگه بیش از `VISUAL_DIFF_WARN_PERCENT` درصد پیکسل‌ها فرق کرده باشن (دیفیس، خرابی لی‌اوت، صفحه‌ی سفید) هشدار می‌ده
- LCP و CLS رو با تزریق `PerformanceObserver` توی همون صفحه اندازه می‌گیره (بدون نیاز به Lighthouse سنگین) و افت محسوس سرعت رو گزارش می‌ده
- انقضای دامنه رو با whois چک می‌کنه (نیاز به دسترسی خروجی TCP:43 از VPS داره؛ بعضی فایروال‌ها این پورت رو می‌بندن)

## قابلیت‌های سبک UptimeRobot

- **مکث/ادامه**: هر سایت رو می‌تونی از صفحه‌ی جزئیاتش موقتاً pause کنی (مثلاً حین دیپلوی) — هیچ چکی انجام نمی‌شه و هشداری نمیاد تا دوباره ادامه بدی.
- **مانیتورینگ کلیدواژه**: تو فرم ویرایش سایت، یه کلیدواژه بذار و انتخاب کن «باید توی صفحه باشه» (مثلاً دکمه‌ی افزودن به سبد) یا «نباید باشه» (مثلاً متن ارور دیتابیس) — اگه شرط نقض بشه، حتی با HTTP 200 هم سایت را down گزارش می‌شه.
- **درصد آپ‌تایم ۷/۳۰/۹۰ روزه**: تو صفحه‌ی هر سایت، بالای نمودارها.
- **صفحه‌ی وضعیت عمومی**: از تنظیمات یه لینک بدون-رمز می‌گیری (`/#/status/<token>`) که فقط سایت‌های «نمایش عمومی»شون روشن رو نشون می‌ده — برای فرستادن به مشتری/کارفرما. لینک رو می‌شه هر وقت خواستی از نو ساخت (لینک قبلی از کار می‌افته).

## تثبیت پایه (v2.0)

- **تست خودکار**: `npm test` (Node's built-in test runner، بدون وابستگی اضافه) + GitHub Actions CI روی هر push/PR
- **لاگ ساختاریافته**: `data/logs/YYYY-MM-DD.log` (JSON، هر روز یه فایل جدا، پاک‌سازی خودکار بعد از ۳۰ روز)
- **بکاپ خودکار دیتابیس**: هر روز ساعت `BACKUP_HOUR` (پیش‌فرض ۴ بامداد) از `data/monitor.db` می‌ره تو `data/backups/`، ۱۴ روز نگه داشته می‌شه
- **`GET /api/health`**: بدون نیاز به رمز، برای مانیتور کردن خودِ سرور از بیرون (نسخه، uptime، تعداد سایت، آخرین چک)
- **تایید دومرحله‌ای (2FA)**: از تنظیمات فعال کن — با Google Authenticator/Authy، حتی اگه رمز پنل لو بره وارد نمی‌شه

## اینسیدنت‌منیجمنت (v3.0)

- **پنجره‌ی تعمیرات**: تو صفحه‌ی هر سایت — یه بازه‌ی زمانی تعریف کن (برای همون سایت یا همه‌ی سایت‌ها)، تو اون بازه هشدار تلگرام نمیاد (خودِ چک‌ها و تایم‌لاین عادی ادامه پیدا می‌کنن) — برای دیپلوی/آپدیت بدون هشدار کاذب
- **مانیتور پورت (TCP)**: تو صفحه‌ی هر سایت، هر تعداد host:port دلخواه (دیتابیس، SSH و ...) رو اضافه کن، مثل بقیه‌ی چک‌ها وضعیتش دنبال می‌شه و تغییرش هشدار می‌ده
- **گزارش SLA**: دکمه‌ی دانلود CSV تو صفحه‌ی هر سایت (۳۰ یا ۹۰ روزه) — لیست اینسیدنت‌های قطعی با مدت‌زمان دقیق

## اقدامات از راه دور (v4.0) — خاموش پیش‌فرض

از داشبورد می‌تونی مستقیم آپدیت پلاگین/پوسته/هسته و پاک‌کردن کش رو روی سایت‌های وردپرسی‌ای که agent روشون نصبه بزنی. **این قابلیت پیش‌فرض کاملاً خاموشه** — تا از تنظیمات صریحاً روشنش نکنی (با یه تاییدیه‌ی دوم)، نه پنل اجازه‌ی صف‌کردن دستور می‌ده و نه agent چیزی دریافت می‌کنه، حتی اگه یه دستور قبلاً صف شده باشه.

وقتی روشنه:
- صفحه‌ی هر سایت (اگه agent وصل باشه) لیست آپدیت‌های موجود پلاگین/پوسته/هسته رو نشون می‌ده، با دکمه‌ی آپدیت تک‌تکشون
- دکمه‌ی «پاک کردن کش» — به‌صورت best-effort با WP Rocket، W3 Total Cache، WP Super Cache، LiteSpeed Cache، SiteGround Optimizer، یا `wp_cache_flush()` عادی
- agent هر ۵ دقیقه دستورهای در صف رو چک می‌کنه، اجرا می‌کنه، نتیجه رو برمی‌گردونه — تاریخچه‌ی کامل تو همون صفحه

**هشدار امنیتی/ریسک واقعی**: یه آپدیت ناسازگار می‌تونه سایت رو خراب کنه. این قابلیت رو فقط اگه واقعاً بهش نیاز داری روشن کن.

## قابلیت‌های آژانسی (v5.0)

- **گروه‌بندی سایت‌ها بر اساس مشتری/پروژه**: هر سایت یه فیلد «مشتری» اختیاری داره؛ لیست سایت‌ها خودکار بر همون اساس گروه‌بندی می‌شه و یه فیلتر بالای صفحه هم هست
- **برندسازی (White-label)**: تنظیمات → اسم و لوگوی پنل رو با برند خودت یا مشتریت عوض کن — رو صفحه‌ی لاگین و هدر همه‌جا اعمال می‌شه
- **مستندات API**: [`docs/API.md`](docs/API.md) — همه‌ی endpointها با نوع auth هرکدوم

چندکاربره‌بودن با نقش (ادمین/مشتری محدود) و صورت‌حساب/پلن عمداً ساخته نشدن — نیاز به عوض‌کردن مدل احراز هویت فعلی (یه رمز مشترک) دارن که یه تصمیم معماری جداست.

## آسیب‌پذیری (CVE) — فاز A از v2

هر پلاگینی که agent گزارش می‌ده با یه دیتابیس آسیب‌پذیری مطابقت داده می‌شه؛ اگه نسخه‌ی نصب‌شده توی بازه‌ی آسیب‌پذیر باشه، تو صفحه‌ی «آسیب‌پذیری‌ها» و صفحه‌ی هر سایت نشون داده می‌شه و برای شدت ≥ `VULN_ALERT_MIN_SEVERITY` هشدار تلگرام (تاپیک امنیت) می‌ره.

- **local-first**: چون هیچ فید JSON رایگانِ بدون‌کلید معتبری برای آسیب‌پذیری وردپرس باقی نمونده (Patchstack صفحه‌ی HTMLه، Wordfence حالا کلید می‌خواد)، هسته‌ی این قابلیت یه دیتابیس محلی دستیه: `data/local-vulnerabilities.seed.json` (موقع بوت لود می‌شه). این‌جاست که آسیب‌پذیری پلاگین‌های **بازار ایران** رو می‌ذاری — مزیتی که MainWP/ManageWP ندارن.
- **افزودن دستی از پنل**: صفحه‌ی آسیب‌پذیری‌ها → «افزودن دستی».
- **فید خارجی اختیاری**: اگه یه‌روز اکانت رایگان Wordfence/Patchstack گرفتی، `EXTERNAL_VULN_FEED_URL` و `EXTERNAL_VULN_FEED_KEY` رو ست کن (فرمت: JSON آرایه به شکل داخلی).
- بازه‌ی نسخه‌ها: `<= 5.5`, `>= 3.6.0 < 3.6.3`, `1.2.*` و... — با یه matcher سبک بدون وابستگی (`server/vuln/versionRange.js`) که نسخه‌های ناقص وردپرسی رو هم هندل می‌کنه.
- وقتی پلاگین آپدیت شد و از بازه خارج شد، مورد خودکار resolved می‌شه.

## Fleet Learning / Update Guard — فاز B از v2

وقتی یه پلاگین روی سایت اول آپدیت شد و باعث مشکل شد، سیستم خودکار **همون آپدیت دقیق رو روی بقیه‌ی سایت‌ها hold می‌کنه**. کاملاً دیترمینیستیک، بدون AI.

- وقتی agent یه رویداد `plugin_update` گزارش می‌ده، یه ارزیابی با تأخیر توی صف پایدار (`pending_verdicts` تو دیتابیس — ری‌استارت گمش نمی‌کنه) ثبت می‌شه؛ یه cron هر ۵ دقیقه ردیف‌های سررسیده رو بررسی می‌کنه.
- **قضاوت فقط با سیگنالی که واقعاً با فرکانس بالا داریم**: چک‌های آپ‌تایم (هر ۵ دقیقه). پنجره‌ی ۱ ساعت بعد آپدیت با ۱ ساعت قبلش مقایسه می‌شه:
  - **bad**: چک ناموفق/۵۰۰ توی پنجره‌ی بعد که قبلش نبود
  - **suspicious**: میانگین پاسخ بیش از ۲ برابر شد
  - **safe**: هیچ‌کدوم
- verdict روی upgrade path (`slug` + `from`→`to`) تجمیع می‌شه. اگه bad/suspicious بود، برای هر سایت دیگه‌ای که همون نسخه‌ی `from` رو داره یه hold ساخته می‌شه.
- agent (پلاگین وردپرسی) قبل از نمایش آپدیت، `/api/update-check` رو صدا می‌زنه؛ اگه hold بود، تو صفحه‌ی افزونه‌های WP-Admin یه بنر هشدار نشون می‌ده (فقط هشدار، مکانیکی جلوی آپدیت رو نمی‌گیره).
- داشبورد: بنر «هشدارهای Fleet Learning» تو صفحه‌ی اصلی + بخش «آپدیت‌های hold‌شده» تو صفحه‌ی هر سایت با دکمه‌ی «آزاد کردن».
- **بدون auto-apply**: این فاز فقط جلوی آپدیت‌های خراب رو می‌گیره، هیچ آپدیتی رو خودش نمی‌زنه.
